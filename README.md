## 三、基准（合规地标准）

你要"把案例作为标准、判断我们重排的作品是否合格"——这点已实现，但有两个必须讲清的设计：

**1) 版权：只取数值，不收图文。** 原书的扫描页与注释受版权保护，不打包进可分发的 skill。
`derive_benchmark.py` 在**你本地的原书副本**上测量，只产出**客观数值**（灰度均匀度、墨色密度等——
属事实数据，非受保护表达），写进 `benchmarks/book_benchmark.json`。原书留在你手里当"真尺子"，
skill 里只有数值和脚本。我已用你上传的书跑出一版基准（5 个干净单栏正文页，中央裁切、按密度过滤）。

**2) 跨媒介不可直接比（重要诚实结论）。** 原书是扫描件、我方是数字直出；同一段灰度，扫描噪声会
把 CV 抬高。实测：原书正文页 CV 中位≈0.40、合格线≈0.46；我方数字直出≈0.24。**这不代表我们"更好"，
而是两者量纲不同。** 所以正确做法是 `compare_to_benchmark.py` 的**同镜对照**：把我方渲染**降采样到扫描
分辨率并加同等模糊/噪声**，再用同一套"眯眼"度量去比。ex01 同镜后 CV=0.144 ≤ 书内合格线 0.46，
密度在带内 → `PASS_vs_book = true`。这条对照已接进 `run_pipeline.py`（正文档位自动执行；诗等 verse 档不适用）。

> 也就是说：**原书负责"设计参数"与"同镜灰度基准"两件事**；至于版心比例，扫描自动量太吵，
> 故不从扫描自动推导，改为按设计显式设定（canon）。

## 四、技术说明（为何这样选）

- **引擎=Pillow 确定性摆字**：容器无 Typst；Noto CJK 是 CFF/OTF，reportlab 无法嵌入。
  Pillow 能直接驱动 OTF，并允许**逐字按网格坐标摆放**——恰好落实"法度=代码"，
  且同一张栅格直接喂给测量"眼睛"。代价：v1 输出为高分辨率栅格 PDF（矢量见路线图）。
- **测量=squint**：对反相图按行高尺度做高斯模糊（模拟眯眼），限定到真实文本块（半空页不被冤枉），
  用稳健离散度（MAD/median）得 robust CV。

## 五、一句诚实的话（与手册一致）

补齐以上，一个零经验的人能**稳定**产出干净、克制、经得起久看、且每个决定都说得清的版面——
这已踩在手册为"入门"划的那条线（可解释）上。但"**和白井一样标准**"最深的那一半——
练出来的眼、策展级品味、对历史气质的判断——正是手册说**任何手册都传不了的**。
本 skill 用**测量**逼近"眼"，用**范例画廊+评分细则**对齐"品味"，但剩下那点 taste 是养出来的、
不是规格写死的。所以目标定为：**始终优秀、完全可解释、忠于白井方法**；
白井级品味靠范例不断逼近、而非保证达到。**范例画廊的质量，基本决定你能逼近到多近。**
